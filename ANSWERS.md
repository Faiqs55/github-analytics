# Technical Assessment Answers — Git-Health CLI

### 1. How to Run
To run this project on a fresh machine:
1. Run `npm install` inside the project root folder to download the single interface styling dependency (`picocolors`).
2. Run `npm start -- <username>` (replacing `<username>` with a real public GitHub account name, e.g., `npm start -- octocat`).
3. To scale the rate limit configuration up to 5,000 queries per hour, prepend a personal access token: `GITHUB_TOKEN=your_token_here npm start -- <username>`.

### 2. Stack Choice
* **Why this stack was chosen:** I chose to build this as a Node.js command-line interface (CLI) using CommonJS syntax and Axios for network connectivity. For an API aggregation tool, building a full graphical web interface introduces massive overhead spent on CSS styling, layout responsiveness, and client-side hosting. A Node.js runtime leverages asynchronous, non-blocking I/O natively, enabling me to handle rapid, structural sequential operations seamlessly while keeping the final implementation ultra-lightweight and dependency-lean.
* **What would have been a worse choice:** A heavy front-end application architecture (such as a full React/Vite layout or Next.js). Introducing client-side hydration, state management overhead, and UI compilation rules under a strict 48-hour deadline would distract entirely from writing bulletproof error handling, native file writing modules, and robust API failure boundaries.

### 3. One Real Edge Case
* **The Edge Case:** The "Zero-Commit / Empty Repository" Arithmetic Guard.
* **File and Line Location:** `src/auditor.js`, Line 43.
* **Explanation:** When a developer creates a public repository on GitHub but has never pushed code or committed files to it, hitting the repository languages endpoint (`/repos/{owner}/{repo}/languages`) returns an empty JSON payload (`{}`). If the code tries to loop through this response without an explicit safety checkpoint, the global byte aggregator counter will sit at exactly `0`. When the code later calculates individual stack distribution percentages (`(bytes / totalGlobalBytes) * 100`), JavaScript will attempt to divide by zero. This causes the application to output corrupt values like `NaN%` across metrics and crash the Markdown formatting script. My code implements an explicit guard statement checking if `totalGlobalBytes > 0` before entering the division loops, completely resolving the edge case.

### 4. AI Usage
I utilized an AI coding model to rapidly scaffold structural code components to optimize speed during this 48-hour project timeline:
* **Tool Used:** ChatGPT (OpenAI)
* **Prompt Provided:** *"Write a JavaScript helper function that calculates the precise number of days that have passed between an ISO 8601 date string and today's date."*
* **Output Received:** A raw math function utilizing `Math.floor()` to divide standard millisecond timestamps.
* **What I Changed and Why:** The output generated by the AI did not handle inputs gracefully if an invalid date format or null value was passed. I refactored it into a clean helper method named `calculateDaysAgo()` within `src/auditor.js`, wrapped the date constructor manually, and integrated it cleanly into our custom repository schema mapping so it could calculate project staleness thresholds without throwing runtime syntax exceptions.

### 5. Honest Gap
* **The Present Vulnerability:** Lack of a client-side local caching layer.
* **Impact:** Because of the tight 48-hour delivery timeline, the tool performs live, remote network operations hitting GitHub infrastructure every single time a user executes the command line. If an assessor tests the exact same large repository list three times consecutively, they risk exhausting the default public API quota unnecessarily.
* **The Extension Strategy with 24 Hours More:** If I had an extra day, I would build a file-system cache broker inside a local `.cache/` folder using Node's `fs` modules. Each successful profile audit would save its parsed payload alongside a 15-minute Time-To-Live (TTL) unix timestamp epoch. Future execution runs targeting the exact same user within that 15-minute window would pull calculations instantly from local storage, dropping network overhead to zero.
